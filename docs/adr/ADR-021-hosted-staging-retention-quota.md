# ADR-021: Hosted Staging Retention and Quota Policy

## Status
Accepted for MVP

## Context
ADR-010 selects cloud-staging-first iPhone ingest for MVP, and ADR-019 allows future `hosted-staging` execution manifests. Existing docs intentionally left exact hosted-storage retention and quota values open.

That is now a material implementation decision because an iOS upload endpoint and staging API cannot be implemented safely without bounded size, retention, and status semantics.

## Decision
Hosted staging is a temporary convenience layer for iPhone-originated uploads. It is not an archive-primary, archive-replica, restore source of record, or cleanup-safety signal.

MVP hosted-staging limits:
- maximum object size: 2 GiB,
- maximum pending staged bytes per library: 25 GiB,
- maximum pending staged objects per library: 500,
- upload reservation expiry: 24 hours if the mobile client never completes the upload,
- completed staging retention: 7 days after upload completion,
- post-archive retention: staging may become cleanup-eligible 24 hours after verified archive-primary placement, verified archive-replica placement, and restore evidence exist for the asset.

Rules:
1. Upload completion may create only a staged/mobile-ingest status. It must not mark an asset archived, verified, restore-ready, cleanup-eligible, or safe to remove from the phone.
2. Hosted staging object identifiers must be opaque ids or object keys generated by the control plane/storage abstraction. They must not contain local device paths, Photos library local identifiers that are unsafe to expose, or user-entered filenames as authority.
3. The control plane may persist staging metadata, quota counters, status, safe error classes, checksums, object size, and correlation ids.
4. Staging retention deletes must be explicit, observable retention work with audit/log output. They must not be hidden automation and must not delete phone originals or user archive copies.
5. If quota is exceeded, the API must block new upload reservations with a safe error and leave existing staged objects untouched.
6. Local development may use filesystem or MinIO-compatible staging only as a storage abstraction behind the same retention/quota rules. Production may use S3-compatible object storage or an equivalent VPS-mounted object-store service, but the VPS must not become the primary archive for originals.
7. Desktop archive jobs may consume `hosted-staging` only after a fetch/source resolver is implemented that enforces object expiry, checksum verification, and safe error reporting.

## Consequences
### Positive
- The iOS app can show uploaded/staged/archiving/verified/blocked states without overclaiming backup safety.
- Hosted storage cost and blast radius are bounded for MVP.
- The policy preserves local-first originals and keeps cleanup separate from upload success.

### Trade-offs
- Very large videos over 2 GiB are blocked in MVP and need a later large-object or local-network ingest path.
- Staged objects can expire before archive placement if the desktop agent is offline too long.
- A retention worker or explicit retention job is still needed before production use, but it must be observable and policy-scoped.

## Revisit Trigger
Revisit when:
- large-video ingest becomes a primary MVP requirement,
- paid plan packaging changes staging quotas,
- production object storage provider constraints require different multipart upload limits,
- local-network iPhone ingest becomes a Phase 2 implementation item, or
- retention cleanup needs a dedicated background worker ADR.
